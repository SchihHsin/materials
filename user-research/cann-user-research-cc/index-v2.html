三个结构性错位

真正的断层不仅是「工具难用」或「文档缺失」，而是底层的抽象模型、生态路径与组织阵型同时出现了错位。透过表象看清盘古训练场景的系统性顽疾。

根因 2.1 · ROOT CAUSE
第二章
The Leaky Abstraction

架构与抽象错位
从透明白盒
到过度暴露硬件

CUDA 的伟大在于 SIMT 抽象——用写串行的思维写并发，显存调度与线程束对开发者相对透明。昇腾基于达芬奇架构（DSA / 张量级指令）追求极致计算密度，代价是抽象层太薄。

硬件细节直接暴露
系统强迫开发者直面 L1/L0 缓存、HBM 搬运、异步 DMA 调度——本该由编译器完成的软硬协同，转嫁给了人力。
Tiling 切分成为人的负担
这正是 Tiling 切分如此困难的原因：高端编译器（LLVM/MLIR）该做的事被推给了算子工程师。
MS debug 高优化失效
底层编译器进行剧烈的图重写与循环展开，源码与机器码的映射完全断裂。
「ASC 其实最困难的一点就是没法调试……说是有 MS debug，但是其实我们这边很少有人用」
王工 · ASC 算子开发
根因 2.2 · ROOT CAUSE
第二章
Cathedral vs Bazaar

生态演进路径
封闭大教堂 vs 开源集市

NVIDIA 是典型的集市模式，十几年自下而上发展，工具链在无数外部开发者捶打下成熟。昇腾生态带有强烈的大教堂色彩——自上而下为打赢盘古大模型战役而催熟。

充满「内部视角」
工具链设计服务于内部定点版本交付，而非第三方开发者。
Host / Device 未解耦
460+ .o 文件冗余、不支持增量编译，说明构建系统没做到宿主机与设备端逻辑的有效解耦。
牺牲了模块化与敏捷
为满足内部交付，牺牲了对第三方极其重要的「模块化插拔」与「敏捷试错」能力。
「实际上我觉得不是很好找，就是东西可能散落在各处」——内部视角下，资产从未为第三方开发者组织过
范工 · 算子需求分析师
根因 2.3 · ROOT CAUSE
第二章
Conway's Law

组织与流程的
康威定律投影

系统的架构受制于产生它的组织沟通结构。大模型训练本质是「数学（算法/Loss）」与「物理学（硬件/显存/算力）」融合的端到端工程，但组织依然是传统的流水线分工。

归因链层层甩锅
Loss 不降时——算法怪算子精度、算子怪框架图优化、框架怪硬件指令异常。
缺乏全栈「性能与精度契约」
没有跨越全栈的 Contract，问题在不同 Team 之间的信息转换成本极高。
「框架进步了但有些功能还没支持好，学习率要是调不准，会出现不收敛」——问题落在算法与框架的边界上，无人认领

三层重构方案

不能再用「修修补补」的战术，必须在工具、资产、流程三层进行降维打击：从面向过程调试转向 IR 与沙箱、从死文档转向可执行活资产、从流水线甩锅转向全栈性能契约。

方案 3.1 · TOOLCHAIN
第三章
From Process Debug → IR & Sandbox

工具层
基于中间表示与沙箱验证

这是解决 30 分钟编译和无法调试的核心：彻底解耦 Host 与 Device，把调试上移到编译器中间层。

解耦 Host / Device 编译链
引入真正的增量编译；改 Host 代码绝不触发 Device 重编。基于 Hash 的智能缓存（类 Bazel）消除冗余 .o 文件。
MLIR 级可调试能力
放弃在最高优化汇编里找对应关系，在 Graph IR / 算子 IR 层做可视化调试，验证 Tiling 逻辑正确性。
数学等价性自动化沙箱
同一组 Tensor 在 CPU/GPU 黄金基准与 NPU 上自动运行，直接输出误差热力图，而非十几万行 16 进制。
「改一行代码，它可能要把整个都编一遍……可能 30 分钟到 1 个小时，就这样很耗时」
王工 · ASC 算子开发
方案 3.2 · ASSETS
第三章
From Dead Docs → Executable Assets

开发者资产层
从死文档到可执行活资产

文档永远滞后于代码，大教堂模式下更甚。要把「资产」从模棱两可的说明，变成开箱即跑的真相。

Executable Truth 基准库
为每类算子（Matmul、FlashAttention、Reduce）提供可 git clone && make run 的极简标杆工程，Tiling 参数、内存预取、异步调度开箱即运行。
封装 Triton-to-Ascend 前端
开发者已习惯 GPU 生态，强行扭转心智成本极高。加大对 OpenAI Triton 等上层 DSL 的对接，写 Python 级代码即可下发到 A5。
把内存管理交还系统
L1/L0 内存管理交给系统级 Auto-Tuning，而非依靠人工经验。
「对于一个新手来说，我想知道这个 API 的前因后果——我为什么有这个 API，它一般用在什么场景下？」

方案 3.3 · PROCESS
第三章
Full-Stack Performance Contract

流程层
建立全栈性能契约机制

打破部门墙必须依靠强有力的工程机制，把跨团队甩锅从源头掐断。

设立「全栈系统架构师」
在盘古等重点场景，需要跨越从 Python 脚本到 CANN 驱动的系统架构师，不负责具体特性，只负责端到端的「排查路径」设计。
性能与精度 SLA 交接制
算法与算子交接的不仅是需求文档，而是含输入分布、FP16/BF16 精度下限、Inf/NaN 处理的可执行测试脚本。
脚本跑通才允许合入
单测在本地跑通，算子代码才允许合入主干，从源头掐断后期跨团队扯皮。
「我怎么能快速地确认这个算子是不是支持的？之前我这边是通过算子名字去找代码」——需求与实现的交接今天仍靠人工

面向未来 3–5 年

不能只问「如何把当前工具做得更好用」，而要问「未来的 AI 开发范式还需要这些工具吗？」——从算力经济、AI 演进范式与泛在计算三个视角，做前瞻洞察。

前瞻 4.1 · AI FOR AI
第四章
From Handcraft → Self-Generation

演进范式
AI 驱动的自我生成

让顶尖工程师去计算 L1/L0 缓存的字节对齐，本质是「前 AI 时代」的手工软件工程，未来将是极大的算力与脑力浪费。

RL / 遗传算法 Auto-Tuning 编译
开发者只描述「输入、输出、数学公式」，编译器自带的 AI Agent 在虚拟沙箱中自动试错成千上万次 Tiling，寻找最优内存排布。
「活的数字孪生体」资产
文档变成孪生沙箱环境：提需求「128k 上下文 FlashAttention」，AI 自动生成代码、配好依赖、在孪生 NPU 跑通，直接交付结果。
前瞻 4.2 · BEYOND CUDA
第四章
Win at the IR Layer

生态战略
跨越 CUDA 阴影
决战中间表示

永远「对标 CUDA」就永远是追赶者。未来竞争不再是硬件指令集之争，而是「编译器中间件的定义权」之争。

主导硬件无关 DSL
硬件去中心化下，开发者拒绝为单一芯片写专有算子。战略性投入 MLIR 与 Triton 的昇腾后端，而非让人学 CANN 底层 API。
让底层硬件彻底「隐身」
最高级的体验是「无感」：NPU 与 GPU 只是算力池中的水和电。
强大的 JIT 运行时分发
代码运行时动态感知分配的是 A5 还是其他芯片，自动下发对应机器码。
前瞻 4.3 · RESILIENCE
第四章
Distributed Time-Travel

分布式系统
万亿参数时代的
韧性与可观测

当几万张卡在跑，任何单卡断点调试都不现实。未来训练是横跨多个数据中心的混合异构超级集群，要求极强的系统级韧性。

分布式时光机观测
记录集群全局状态快照。当 Loss 跑飞或某卡出现 NaN，系统像时光机一样自动回滚前 100 个 Step，隔离局部数据流慢放回放。
确定性执行与自动容错
框架提供严格的数学确定性保证：无论硬件拓扑如何变化、机器宕机重启，都像分布式数据库（Spanner）一样保证事务级一致与结果绝对可复现。
