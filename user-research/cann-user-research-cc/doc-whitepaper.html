王工 · ASC 算子开发

代号：王工。5 年以上嵌入式/底层开发经验，技术栈包括 C/C++、CUDA 迁移经验和汇编基础，当前职责是昇腾 Ascend C 算子开发与优化。

  • 目标任务：开发高效、可复用的算子实现；性能达到或接近 CUDA 业界水平；快速定位和解决算子功能/性能问题。
  • 痛点：调试困难、编译慢、日志难读、文档缺失；关键词是调试 efficiency、编译时间、算子性能、Tiling 切分。
ASC 其实最困难的一点就是没法调试…A5 上说是有 MS debug，但是其实我们这边人很少有人用。改一行代码，它可能要把整个都编一遍…可能 30~1 个小时。

陈工 · 高校训练

代号：陈工。身份是高校副教授/研究员，研究方向是大模型训练优化、数据增强，技术栈包括 PyTorch、CUDA、transformers，团队规模 10+ 研究生。

  • 目标任务：探索昇腾平台训练能力，评估迁移可行性；指导团队训练实践；教学科研结合，培养昇腾生态人才。
  • 痛点：算子开发文档与示例不完整、迁移开发时间长、训练优化与故障排查困难、NPU 与 GPU 精度对齐难。

  • 目标任务：学习大模型训练技术与昇腾平台使用；探索训练优化方法；完成毕业论文/毕业项目。
  • 痛点：学习曲线陡峭、社区支持不足、文档可读性差、配置环境耗时且易出错。

黄工 · 训练 Infra

训练框架 Infra 工程师，承担任务提交、资源排队、训练监控、故障定位和实验管理。扩样本显示，他的痛点把研究从单点开发体验推进到万卡级生产训练成本。

  • 目标任务：支撑大规模训练稳定运行，降低中断和定位成本；管理一天数十条任务并追踪训练状态。
  • 痛点：万卡训练成本高，任务状态不透明，靠手动刷网页，异常无主动通知，实验管理靠脚本和文件。
  • 机会点：训练任务中控台、资源 ETA、IM/邮件通知、异常告警、日志聚合和实验结果图形化对比。
一到两个小时刷一次网页；如果万卡训练出现 AN 或 loss 跑飞，定位效率会直接变成十万到数十万元量级的成本问题。

庞工 · 整网性能分析

整网性能分析工程师，负责 profiling、性能拆解、算子替换验证和底层数据搬运分析。他的反馈补强了昇腾侧与 Nsight/nsys 的工具差距。

  • 目标任务：定位性能瓶颈、分析整网运行效率、评估替换算子是否真正提升性能。
  • 痛点：缺少类似 nsys 的底层数据采集，profiling 对比弱，概率性精度问题难复现，替换算子验证可能跑 1-2 天。
  • 机会点：profiling diff、数据搬运量可视化、rank/算子/step 联动分析、概率问题复现包。
nsys 能采集到很多信息，比如数据搬运量；我们缺少这种数据。

周工 / 王工 · MDE 与算子适配

MDE 与算子适配角色把“算子开发难”进一步拆成详细设计、精度定位、需求澄清、主线发布和自定义算子库节奏问题。

  • 目标任务：完成 ASC 算子详细设计、需求澄清、开发验收和生产替换。
  • 痛点：普通精度问题仍需半天到一天人工逐层打印；需求澄清 2-4 周；主线算子开发至少 1 个月，发版跟随 3 个月节奏。
  • 机会点：单一帮助入口、精度对比工具、自定义算子库规范化、Triton 到 AscendC 下沉路径。
需求澄清最快两周，踩平到开始开发至少三周到一个月；自定义算子库约 15 天发版，节奏明显不同。
Journey · 端到端链路

全场景旅程的痛点从“接入准备”一路延伸到“部署验证”

盘古训练体验包含十个阶段。早期的问题是技术选型和环境准备，中段集中在算子开发、编译和调试，训练后段则转向监控、精度对齐、测评和部署。

01 需求调研缺少技术选型指南和算子支持查询。
02 环境搭建MS debug 配置复杂，缺少一键验证。
03 算子开发Tiling、内存和异步调度缺少方法论。
04 编译构建增量编译缺失，等待时间长。
05 调试定位高优化等级下源码与汇编映射断裂。
06 数据准备数据清洗、抽检和基准数据集获取成本高。
07 训练启动分布式配置、混合精度和 checkpoint 策略需要手工处理。
08 训练监控loss、设备利用率、NPU 系统日志缺少统一视图。
09 验证测试缺少自动化基准对比和统一测评工具。
10 推理部署模型转换、多端部署和灰度能力不成体系。

Root Causes · 深层根因

真正的系统性问题来自抽象、生态和组织三重错位

真正的断层，不仅仅是工具难用或文档缺失，而是底层的抽象模型、生态路径与组织阵型出现了错位。报告从组织架构与流程以及昇腾与 CUDA 生态对比的宏观视角，系统性输出改进框架与具体建议。

架构与抽象错位

从“透明白盒”到“过度暴露硬件细节”的退化。表象痛点是 Tiling 切分难、MS debug 优化后失效、编译耗时 30-60 分钟。

CUDA 的成功抽象是 SIMT，开发者可以用写串行代码的思维去写并发；昇腾底层基于达芬奇架构（DSA/张量级指令），追求计算密度的代价是抽象层太薄，系统强迫算子开发者直接面对 L1/L0 缓存、HBM 搬运、异步 DMA 调度等底层硬件细节。

Leaky Abstraction软硬协同外溢

生态演进路径错位

封闭的“大教堂”与开源的“集市”之间存在差异。NVIDIA 生态经历十几年自下而上发展，Nsight 等工具链是在外部开发者持续使用中成熟的。

昇腾生态带有内部视角：编译产物冗余（460+ 个 .o 文件）和不支持增量编译说明底层构建系统没有做到 Host 与 Device 逻辑有效解耦，牺牲了第三方开发者重视的模块化插拔和敏捷试错。

大教堂模式敏捷试错不足

组织与流程错位

组织与流程的“康威定律”投影体现为物理学与软件工程的摩擦。系统架构受组织沟通结构影响，而盘古训练是融合数学（算法/Loss）与物理学（硬件/显存/算力）的端到端工程。

现有流水线分工容易导致 Loss 不降时相互归因：算法认为算子精度不够，算子认为框架图优化出错，框架认为硬件底层指令异常。缺少跨越全栈的性能与精度契约。

康威定律全栈契约缺失
根因判断：要解决盘古训练体验问题，不能只补文档或修单点工具，必须同时重构工具反馈闭环、开发者资产形态和跨团队交付契约。

Future · 面向未来

未来 3-5 年的目标不是更会调试，而是减少手工调试

如果把目光投向未来 3 到 5 年甚至更远的大模型算力网络，关键问题不是如何把当前工具做得更好用，而是未来的 AI 开发范式还是否需要这些工具。面向万亿参数训练和异构算力池，盘古生态需要从手工微雕式算子开发转向 AI for AI、硬件无关 IR 和分布式可观测性。

演进范式：AI 驱动的自我生成

从“手工微雕”走向“AI 驱动的自我生成 (AI for AI)”。当前最大痛点，如 Tiling 切分难、汇编级调试、性能调优耗时长，本质上是前 AI 时代的手工软件工程模式。

未来的算子开发应引入基于强化学习或遗传算法的 Auto-Tuning 编译器前端。开发者只描述输入、输出和数学公式，AI Agent 在虚拟沙箱中成千上万次试错 Tiling、内存排布和执行流。面向“支持 128k 上下文的 FlashAttention”这类需求，系统应自动生成框架代码、配置依赖边界，并在孪生 NPU 模拟器中跑通性能数据。

分布式系统：韧性与可观测性

当前痛点多集中在单卡、单算子调试上。随着盘古大模型走向万亿乃至十万亿参数，训练会横跨多个数据中心和混合异构算力，系统级韧性会成为核心能力。

调试应从单点断点升级为分布式时光机观测。当几万张卡在跑时，工具需要记录全局状态快照；Loss 跑飞或某台 NPU 出现 NaN 异常时，自动回滚到前 100 个 Step，将局部数据流隔离并慢放回放。框架还需要提供严格数学确定性，无论硬件拓扑变化或机器重启，都保证运算结果可复现。
