# 盘古训练场景用户研究报告

> **报告版本**: v2.0（基于服务设计方法论）
> **生成日期**: 2026 年 6 月 9 日
> **研究者**: 用户研究团队
> **数据来源**: 访谈录音转文字（8 份）+ 问题清单 Excel（155+ 用户）

---

# 执行摘要

本报告采用**服务设计（Service Design）**系统方法，对盘古训练与算子开发场景的用户体验进行全面调研分析。通过多源用户研究数据的三角验证，识别核心体验问题，形成可落地的改进框架。

## 📊 研究概况

| 维度 | 数据 |
|------|------|
| **访谈对象** | ASC 算子开发（2 人）、PyPTO 开发（2 人）、高校 GPU 训练（3 人）、微调场景（1 人） |
| **问题清单** | 172+ 记录，覆盖 25+ 用户反馈 |
| **访谈时长** | 累计 8 小时+，逐字稿 800K+ 字符 |
| **分析方法** | 服务设计四要素：角色→旅程→触点→洞察 |

# 第一部分：用户画像与干系人地图

## 1.1 核心角色识别与服务对象

基于调研数据，盘古训练场景涉及以下核心角色生态系统：

### 📌 研发侧角色

| 角色 | 定义 | 职责范围 | 典型痛点 |
|------|------|----------|----------|
| **ASC 算子开发工程师** | 使用 Ascend C/SC语言开发底层算子 | 自定义算子开发、性能优化、算子调试 | 「ASC 最困难的一点就是没法调试」- 王文满 |
| **PyPTO 算子开发工程师** | 使用 PyPTO 框架开发算子 | 算子封装、Python 接口开发 | 开发流程复杂，文档缺少 |
| **算子需求分析师** | 负责算子需求收集和梳理 | 需求收集、优先级评估、需求文档编写 | 「我怎么能快速找到这个算子是不是支持」- 范汇华 |
| **框架开发工程师** | 负责训练框架开发、维护 | 昇腾框架适配、算子库开发、工具链建设 | 接口设计不够直观 |

### 📌 训练侧角色

| 角色 | 定义 | 职责范围 | 典型痛点 |
|------|------|----------|----------|
| **GPU 模型训练工程师** | 使用 GPU 进行模型训练的开发者 | 模型设计、训练代码实现、性能调优 | NPU 与 GPU 训练精度对齐困难 |
| **A5 模型训练工程师** | 使用昇腾 A5 芯片进行训练 | 昇腾环境下的模型训练、适配 | 工具链不如 CUDA 成熟 |
| **模型训练微调工程师** | 负责模型的微调工作 | SFT、RL 等微调场景 | 缺少一键式配置 |

### 📌 高校/研究机构角色

| 角色 | 定义 | 职责范围 | 典型痛点 |
|------|------|----------|----------|
| **高校教师/教授** | 高校研究院系带头人 | 学术研究、团队指导、教学 | 「算子开发文档与样例不足」- 浙大陈博宏 |
| **高校学生/研究生** | 高校模型训练研究者 | 模型开发、学术研究、技术探索 | 学习曲线陡峭、社区支持不足 |

### 📌 平台侧角色

| 角色 | 定义 | 职责范围 | 典型痛点 |
|------|------|----------|----------|
| **云平台运维工程师** | 负责云平台 NPU/GPU设施运维| 作业调度、性能监控、资源维护 | 「作业调度与管理缺乏统一视图」- 周丽洒 |

## 1.4 角色统计与占比分析

基于问题清单 Excel（155 用户样本）的统计分析：

| 角色分类 | 用户数量 | 用户占比 | 问题数量 | 问题密度 |
|----------|----------|----------|----------|----------|
| **ASC 算子开发工程师** | 8 | 5.2% | 54 | 高 |
| **PyPTO 算子开发工程师** | 6 | 3.9% | 26 | 中 |
| **算子需求分析师** | 4 | 2.6% | 11 | 中 |
| **GPU 模型训练工程师** | 13 | 8.4% | 73 | 高 |
| **A5 模型训练工程师** | 4 | 2.6% | 12 | 低 |
| **模型训练微调工程师** | 1 | 0.6% | 10 | 低 |
| **云平台运维工程师** | 1 | 0.6% | 5 | 低 |
| **高校教师/教授** | 13 | 8.4% | 73 | 高 |
| **高校学生/研究生** | 1 | 0.6% | 1 | 极低 |
| **其他/未分类** | 104 | 67.1% | - | - |

**关键发现**：
- 核心研发角色（ASC 算子开发、GPU 训练、高校教师）贡献了约 **30%** 的反馈
- 问题密度最高的角色是 **ASC 算子开发工程师**（人均 6.75 个问题）
- 开发者资产类问题（文档、工具）占问题的 **70%+**

#改进框架
从组织架构与流程以及昇腾与CUDA生态对比的宏观视角，为您深度剖析盘古训练场景的底层逻辑，并系统性地输改进框架与具体建议。
一、 深度洞察：透过表象看清盘古训练场景的“系统性顽疾”
真正的断层，不仅仅是“工具难用”或“文档缺失”，而是底层的抽象模型、生态路径与组织阵型出现了错位。

1. 架构与抽象错位：从“透明白盒”到“过度暴露硬件细节”的退化
表象痛点：Tiling 切分难、MS debug 优化后失效、编译耗时 30-60 分钟。

深度根因（The Leaky Abstraction）：CUDA 的伟大在于它提供了一个极具欺骗性但也极其成功的抽象——SIMT（单指令多线程）。开发者可以用写串行代码的思维去写并发，底层硬件的显存调度和线程束（Warp）对开发者是相对透明的。

昇腾的困局：昇腾底层基于达芬奇架构（DSA/张量级指令），这种架构天然追求极致的计算密度，但其代价是抽象层太薄。系统强迫算子开发者直接面对 L1/L0 缓存、HBM 搬运、异步 DMA 调度等极度底层的硬件细节。这就解释了为什么 Tiling 切分如此困难——框架把本该由高端编译器（如 LLVM/MLIR）完成的“软硬协同”工作，转嫁给了人力。MS debug 在高优化下失效，正是因为底层编译器进行了剧烈的图重写和循环展开，导致源码和机器码的映射完全断裂。

2. 生态演进路径：封闭的“大教堂” vs 开源的“集市”
表象痛点：文档散乱、自定义算子包测试困难、无法直接对接 Triton。

深度根因：NVIDIA 生态是典型的“集市模式”，经历了十几年的自下而上发展，工具链（如 Nsight）是在无数外部开发者的捶打下成熟的。而昇腾生态（包括 CANN 和 MindSpore）带有强烈的“大教堂模式”色彩，它是自上而下为了打赢大模型战役（如盘古）而催熟的。

结果：这导致工具链的设计充满了“内部视角”。例如，编译产物冗余（460+个.o文件）和不支持增量编译，说明底层的构建系统（Build System）没有做到宿主机（Host）与设备端（Device）逻辑的有效解耦。系统为了满足内部定点版本的交付，牺牲了对第三方开发者极其重要的“模块化插拔”和“敏捷试错”能力。

3. 组织与流程的“康威定律”投影：物理学与软件工程的摩擦
表象痛点：跨团队扯皮、NPU/GPU 精度对齐难。

深度根因：康威定律指出，系统的架构受制于产生它的组织的沟通结构。在盘古训练中，大模型训练本质是一个融合了“数学（算法/Loss）”与“物理学（硬件/显存/算力）”的端到端工程，但现有的组织架构依然是传统的“流水线分工”（算法团队 -> 框架团队 -> 算子团队 -> 驱动团队）。

结果：当 Loss 不降时，算法认为是算子精度不够，算子认为是框架图优化出错，框架认为是硬件底层指令异常。缺乏一个跨越全栈的“性能与精度契约（Contract）”，导致问题在不同 Team 之间的信息转换成本极高。

二、 破局方案：系统化的重构与优化路径
要解决上述深度问题，不能再用“修修补补”的战术，必须在资产、工具和流程上进行“降维打击”。

1. 工具层（Toolchain）：从“面向过程调试”转向“基于中间表示（IR）与沙箱验证”
这是解决 30 分钟编译和无法调试的核心。

彻底解耦 Host 与 Device 编译链：引入真正的增量编译。改动 Host 控制代码时，绝不应触发 Device 端张量运算代码的重编。引入基于 Hash 值的智能缓存构建系统（类似 Bazel），消除冗余的 .o 文件生成。

引入 MLIR 级别的可调试能力：放弃在最高优化级别的汇编代码中找对应关系。应该在编译器中间层（如 Graph IR 或算子 IR 层）开发可视化调试工具。让开发者能看到未被极限优化的中间计算流，并在此验证 Tiling 逻辑的正确性。

构建“数学等价性”自动化沙箱：不要让人去阅读十几万行的 16 进制仿真日志。提供一个自动化的工具链，输入一组 Tensor，自动将其同时在 CPU/GPU（黄金基准）和昇腾 NPU 上运行，直接输出误差热力图（Error Heatmap），将黑盒的十六进制数据直接翻译成可视化的精度差异。

2. 开发者资产层（Assets）：从“散落的死文档”转向“可执行的活资产”
文档永远会滞后于代码，大教堂模式下的文档更是如此。

建立“Executable Truth（可执行的真相）”基准库：废弃那些模棱两可的 API 说明。为每一类算子（Matmul, FlashAttention, Reduce 等）提供一个可以直接 git clone && make run 的极简标杆工程。在这个工程里，Tiling 参数、内存预取逻辑、异步调度的正确范式必须是“开箱即运行”的。

封装 Triton to Ascend 编译器前端：既然开发者习惯了 GPU 的生态，强行扭转心智成本极高。战略上必须加大对 OpenAI Triton 等上层 DSL（领域特定语言）的对接，通过编译器后端的努力，让开发者写 Python 级代码就能下发到 A5，把底层的 L1/L0 内存管理交还给系统级的 Auto-Tuning（自动调优）工具，而不是依靠人工经验。

3. 组织流程层（Process）：建立“全栈性能契约”机制
打破部门墙，必须依靠强有力的工程机制。

设立“全栈系统架构师”角色：在盘古训练等重点场景中，不能只有算法 PM 和底层 PM，必须有跨越从 Python 脚本到底层 CANN 驱动的系统架构师。他们不负责具体特性，只负责端到端的“排查路径”设计。

推行“性能与精度 SLA”交接制：算法与算子团队之间交接的不仅是需求文档，必须是一份包含“输入分布、期望 FP16/BF16 精度下限、极端值处理逻辑（如 Inf/NaN 处理）”的可执行测试脚本（Unit Test）。只有该脚本在本地跑通，算子代码才允许合入主干，从源头上掐断后期的跨团队扯皮。

##面向未来
如果我们要把目光投向未来 3 到 5 年（甚至更远）的大模型算力网络，我们不能仅仅思考“如何把当前的工具做得更好用”，而是要思考“未来的 AI 开发范式还需要这些工具吗？”

以下是从算力经济、AI 演进范式、以及泛在计算三个多元视角的深度前瞻洞察：

一、 演进范式视角：从“手工微雕”走向“AI 驱动的自我生成 (AI for AI)”
当前报告中最大的痛点（Tiling 切分难、汇编级调试、性能调优耗时长）本质上是“前 AI 时代”的手工软件工程模式。让顶尖的工程师去计算 L1/L0 缓存的字节对齐，在未来将是一种极大的算力与脑力浪费。

未来的算子开发（Auto-Tuning 与 RL 编译）：
未来的昇腾底层抽象不应该再依赖人去写几十万行的 C++ 或汇编。系统应当引入基于强化学习（RL）或遗传算法的 Auto-Tuning 编译器前端。开发者只需用高层语义描述“输入是什么、输出是什么、数学公式是什么”，编译器自带的 AI Agent 会在虚拟沙箱中自动进行成千上万次的 Tiling 切分试错，寻找最优的内存排布和执行流。

重塑开发者资产——“活的数字孪生体”：
未来的文档不该是静态网页，而是一个昇腾专属的数字孪生沙箱环境。开发者向 AI 提出一个需求（如“我要写一个支持 128k 上下文的 FlashAttention”），AI 自动生成框架代码、配置好依赖边界，并在后台的孪生 NPU 模拟器中跑通性能数据，直接将结果（而非单纯的代码或文档）交付给开发者。

二、 生态战略视角：“跨越 CUDA 阴影”的降维打击，决战中间表示层 (IR)
昇腾如果永远在“对标 CUDA”，就永远处于追赶者的疲惫状态。未来算力生态的竞争，不再是底层硬件指令集的竞争，而是“编译器中间件的定义权”之争。

全面拥抱并主导硬件无关的 DSL（领域特定语言）：
正如你在报告中提到的 Triton，未来的算力生态将呈现“硬件去中心化”。开发者将拒绝为任何单一芯片（无论是 NVIDIA 还是昇腾）编写专有算子。昇腾的未来破局点在于：战略性放弃让开发者直接学习 CANN 的底层 API，转而投入重兵优化 MLIR（多层中间表示）和 Triton 的昇腾后端。

让底层硬件彻底“隐身”：
最高级的开发者体验是“无感”。未来的盘古训练生态，对算法工程师和数据科学家而言，NPU 和 GPU 应该只是算力池中的水和电。通过构建强大的 JIT（即时编译）层，代码在运行时动态感知当前分配的是 A5 还是其他芯片，自动下发对应的机器码。

三、 分布式系统架构视角：面向万亿参数时代的“韧性与可观测性”
当前的痛点多集中在单卡、单算子的调试上。但随着盘古大模型走向万亿乃至十万亿参数，未来的训练场景将是横跨多个数据中心、混合异构算力的超级集群，这就要求极强的系统级韧性。

从“单点调试”升级为“分布式时光机观测 (Distributed Time-Travel Debugging)”：
当几万张卡在跑的时候，任何单卡的断点调试都是不现实的。未来的调试工具需要记录训练集群的全局状态快照。当 Loss 突然跑飞，或者某台机器的 NPU 出现 NaN 异常时，系统能像“时光机”一样，自动回滚到前 100 个 Step，并将出问题的局部数据流隔离开来，在本地或云端的模拟器中进行慢放回放。

确定性执行与自动化容错：
报告中提到“NPU 与 GPU 训练精度对齐是难点”。面向未来，框架需要提供严格的数学确定性保证。无论底层的硬件拓扑如何变化，或者中间有机器宕机重启，模型训练都应当像分布式数据库（如 Spanner）一样，保证事务级的数据一致性和运算结果的绝对可复现。
